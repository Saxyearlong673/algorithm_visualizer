# 数据结构演练场设计文档

- 日期：2026-03-28
- 项目代号：数据结构演练场
- 当前阶段：设计 / Brainstorming 产出
- 目标版本：首版（排序 + 查找动态演示与练习）

## 1. 项目目标

构建一个动态展示数据结构与算法过程的网站，首版聚焦“排序”和“查找”模块。产品定位以“练习平台优先”为核心，不只是播放动画，还要让用户能够通过自定义输入、单步执行、代码高亮和步骤说明，真正理解算法执行过程。

项目首版目标：

1. 清晰展示算法执行过程，让用户能看懂每一步发生了什么。
2. 页面视觉好看、层级清楚，动画服务于理解而不是炫技。
3. 用户可以主动操作和练习，而不是只能被动观看。
4. 为后续扩展链表、栈、队列、树、图和 CRUD 动态演示保留统一架构。

## 2. 产品定位与用户体验目标

### 2.1 产品定位

这是一个面向学习与练习的数据结构算法可视化网站。首版优先满足以下场景：

- 学生学习排序和查找算法
- 面试准备时复习算法过程
- 教学演示时展示关键步骤
- 自己输入数据并观察算法行为

### 2.2 用户体验目标

产品体验应满足以下要求：

- **清晰**：用户能快速识别当前比较元素、交换元素、已排序区域、搜索区间、关键指针。
- **好看**：整体风格具备科技感和现代感，但不牺牲信息密度与可读性。
- **可操作**：用户可以输入数据、控制速度、切换播放模式、重置过程。
- **可学习**：动画、代码高亮和步骤说明始终同步。
- **可扩展**：新增算法时不需要重写整套页面机制。

## 4. 信息架构

### 4.1 页面结构

首版建议采用以下页面结构：

1. **首页**
   - 项目介绍
   - 模块入口（排序、查找）
   - 推荐算法卡片
   - 快速进入最近使用算法

2. **算法练习页**
   - 顶部信息区：算法名称、复杂度、稳定性、适用场景
   - 左侧控制区：输入、随机生成、速度控制、播放控制
   - 中间动态展示区：数组柱状图 / 搜索区间 / 指针状态
   - 右侧代码区：算法代码、当前执行行高亮
   - 辅助说明区：当前步骤解释、统计指标

### 4.2 导航结构

导航先按算法类型组织：

- 排序
- 查找

后续预留扩展入口：

- 线性表与 CRUD
- 栈与队列
- 树
- 图

## 5. 交互设计

### 5.1 核心交互

用户在算法练习页上可执行以下操作：

- 输入自定义数组
- 点击随机生成数组
- 开始自动播放
- 暂停播放
- 单步前进
- 重置到初始状态
- 切换算法
- 调整动画播放速度
- 调整数列规模（在合理上限内）

### 5.2 交互规则

- 切换算法时，保留最近一次输入数组，但重置播放状态。
- 单步执行和自动播放共用同一个步骤序列。
- 当用户输入非法数据时，页面直接提示并阻止执行。
- 二分查找要求输入为有序数组。首版可以采用以下规则之一并在实现阶段定稿：
  - 提示用户必须输入有序数组
  - 或由系统自动生成有序示例数据作为默认输入

本设计当前建议：**默认提供有序示例数据，并在用户输入无序数组时明确提示二分查找需要有序数组。**

## 6. 视觉设计方向

### 6.1 整体风格

建议采用深色科技感风格，强调高对比和清晰层级。

设计原则：

- 动画区是视觉核心
- 控制区简洁直观
- 代码区可读性高
- 颜色服务于状态表达

### 6.2 排序可视化策略

- 使用柱状图表示数组元素值
- 当前比较元素高亮
- 已排序区域使用成功色
- 当前基准值 / 待插入值 / 关键元素使用强调色
- 元素交换采用平滑过渡动画

### 6.3 查找可视化策略

- 顺序查找：逐项扫描高亮
- 二分查找：展示 low / mid / high 指针
- 已排除区间弱化显示
- 当前搜索范围高亮显示

### 6.4 响应式策略

- 首版优先桌面端体验
- 平板可正常使用
- 手机端只保证浏览可用，不承诺完整练习体验

## 7. 技术设计

### 7.1 技术选型

首版建议使用：

- React
- TypeScript
- Vite
- Tailwind CSS

不使用后端，先实现纯前端静态站。

### 7.2 架构分层

#### 页面层
负责页面结构、路由组织和模块切换。

#### 可视化组件层
建议拆分以下核心组件：

- `ControlPanel`
- `ArrayBars`
- `SearchTrack`
- `PointerMarkers`
- `CodeViewer`
- `StepExplanation`
- `MetricsPanel`

#### 算法执行引擎层
该层负责将算法执行过程抽象为统一步骤事件流，是整个系统的核心。

#### 算法定义层
每个算法各自实现逻辑，但统一输出标准步骤结构。

## 8. 步骤事件流设计

### 8.1 设计目标

算法逻辑不直接操作 UI，而是输出一串结构化步骤，交给播放器和可视化层消费。

### 8.2 典型步骤类型

建议事件模型至少支持以下能力：

- `compare(i, j)`：比较两个位置
- `swap(i, j)`：交换两个位置
- `setValue(i, value)`：修改某个位置的值
- `markSorted(i)`：标记某个元素已进入有序区
- `setPivot(i)`：设置快速排序 pivot
- `movePointer(name, index)`：移动指针
- `highlightCode(line)`：高亮代码行
- `message(text)`：展示当前步骤说明
- `setRange(left, right)`：设置当前处理区间
- `markFound(index)`：标记找到目标
- `markDiscardedRange(left, right)`：标记被排除区间

### 8.3 设计原则

- 算法层输出“发生了什么”
- 可视化层决定“怎么显示”
- 播放器决定“什么时候推进到下一步”

这样可以实现算法逻辑、动画表现和页面 UI 的解耦。

## 9. 算法模块组织

建议每个算法以独立模块存在，例如：

- `insertionSortSteps`
- `shellSortSteps`
- `bubbleSortSteps`
- `quickSortSteps`
- `selectionSortSteps`
- `heapSortSteps`
- `mergeSortSteps`
- `radixSortSteps`
- `linearSearchSteps`
- `binarySearchSteps`

每个模块至少提供：

- 算法元信息（名称、复杂度、稳定性、适用场景）
- 代码展示内容
- 输入约束说明
- 步骤生成函数

## 10. 状态管理

首版不引入重型状态管理库。建议使用 React state 或 reducer 管理播放状态。

建议播放器状态包含：

- 当前算法
- 初始输入数据
- 当前数据状态
- 当前步骤索引
- 当前高亮代码行
- 当前说明文案
- 当前统计指标
- 当前速度
- 播放状态（播放中 / 暂停）

## 11. 输入与数据约束

### 11.1 输入方式

- 用户通过逗号分隔输入数组
- 支持随机生成示例数据

### 11.2 校验规则

- 输入必须为数字序列
- 数组长度应限制在适合动画展示的范围内
- 基数排序仅接受非负整数
- 二分查找要求数据有序

### 11.3 建议约束

首版建议将数组长度限制在一个便于视觉展示和性能控制的范围内，例如 5 到 30 个元素。最终数值可在实现计划阶段确认。

## 12. 练习模式设计

首版以练习平台优先，因此除了播放演示，还应保留练习属性。

### 12.1 首版练习能力

- 用户自定义输入
- 单步执行
- 调速观察
- 重复练习
- 本地记忆最近一次输入

### 12.2 第二阶段练习增强

后续可以扩展：

- 预测下一步
- 关键步骤答题
- 错误操作提示
- 算法之间对比练习

这些能力不纳入首版必做范围。

## 13. 分阶段实施建议

### Phase 1：页面和布局骨架
- 首页
- 算法练习页
- 控制面板与展示区布局
- 代码区和说明区基础组件

### Phase 2：统一执行引擎
- 定义步骤事件模型
- 实现播放器机制
- 打通事件流到 UI 展示层

### Phase 3：首批代表算法验证
优先实现：
- 冒泡排序
- 快速排序
- 二分查找

原因：
- 冒泡排序适合验证基础比较和交换动画
- 快速排序适合验证 pivot、区间和递归分治过程
- 二分查找适合验证指针和范围缩减表现

### Phase 4：补齐全部首批算法
接入其余排序和查找算法，统一元信息和说明文案。

### Phase 5：练习增强与细节优化
- 本地持久化
- 轻量练习模式
- 默认案例优化
- 动效和视觉细节优化

## 14. 测试与质量保障

首版应覆盖以下验证重点：

- 不同算法的步骤生成是否正确
- 动画播放与代码高亮是否同步
- 暂停 / 单步 / 重置是否稳定
- 输入非法值时是否有明确提示
- 极端数据规模下页面是否仍可用
- 查找类算法是否正确表现区间变化

建议至少包含：

- 算法步骤生成函数的单元测试
- 播放器状态流转测试
- 核心交互流程的基础 UI 测试

## 15. 关键设计结论

本次 brainstorming 已确认的关键结论如下：

1. 产品定位为**练习平台优先**。
2. 首版覆盖排序和查找模块。
3. 查找首批只做**顺序查找**和**二分查找**。
4. 页面必须支持**真实代码联动高亮**。
5. 技术路线采用**纯前端静态站**。
6. 产品形态采用**单页交互练习平台 + 统一步骤事件流**。
7. 视觉目标为**清晰、好看、突出动态过程**。
8. 设备策略为**桌面优先**。
